Risk Analysis for K > 1

Problem Statement

Our current implementation supports K = 1 risk analysis: it iterates over every reviewer, removes them one at a time, re-runs the max-flow, and checks whether the assignment remains feasible. A natural extension is to consider K > 1: what happens when two or more reviewers become unavailable simultaneously?

Proposed Approach

The following approach extends the existing max-flow pipeline without requiring a fundamentally different algorithm:

  1. Enumerate combinations. For each subset S of K reviewers (there are C(R, K) such subsets), build a reduced reviewer pool by removing S.
  2. Rebuild and re-run max-flow. For each reduced pool, construct the flow graph via buildReviewFlowGraph and run Edmonds-Karp to compute the maximum achievable flow.
  3. Check feasibility. If the achieved flow is less than totalRequiredReviews (P × MinReviewsPerSubmission), the subset S is critical — its absence breaks the assignment.
  4. Report. Return all critical subsets of size K.

Complexity Analysis

Let P be the number of submissions, R the number of reviewers, V = P + R + 2 the number of vertices, and E the number of edges in the flow graph.

Component Complexity
Enumerate C(R, K) subsets O(R^K / K!)
Build flow graph per subset O(P × R)
Edmonds-Karp per subset O(V × E²)
Total O(C(R, K) × (P×R + V×E²))

For K = 2 this is O(R² × V×E²), which is practical for typical conference sizes (R < 100). For K = 3 it becomes O(R³ × V×E²), still tractable. Beyond K = 3 the combinatorial explosion makes exhaustive enumeration impractical.

Optimisation ideas:**

  • Pruning: If removing a single reviewer already breaks feasibility (K = 1 risky reviewer), any superset containing that reviewer is also critical — no need to test further.
  • Early termination: After building the reduced graph, a quick min-cut analysis can detect infeasibility without running the full Edmonds-Karp.

Non-Trivial Input Example

Consider the following scenario (mode 1, primary-only matching):

#Submissions
#Id, Title, Authors, E-mail, Primary, Secondary
1, "Topic A Paper", Alice, alice@x, 1,
2, "Topic B Paper", Bob, bob@x, 2,
3, "Topic A Paper 2", Carol, carol@x, 1,
#Reviewers
#Id, Name, E-mail, Primary, Secondary
1, Rev1, r1@x, 1,
2, Rev2, r2@x, 1,
3, Rev3, r3@x, 2,
4, Rev4, r4@x, 2,
#Parameters
MinReviewsPerSubmission, 2
MaxReviewsPerReviewer, 2

Baseline (all reviewers available):**

  • Submissions 1, 3 (topic 1) match reviewers 1, 2. Submission 2 (topic 2) matches reviewers 3, 4.

  • Total required: 3 × 2 = 6. Achievable: reviewer 1 assigns 2 to subs {1,3}, reviewer 2 assigns 2 to subs {1,3}, reviewer 3 assigns 2 to sub 2, reviewer 4 assigns 2 to sub 2. But sub 2 only needs 2, so flow = 6. Feasible.

    K = 2, remove reviewers {1, 3}:**

  • Remaining reviewers: 2 (topic 1), 4 (topic 2).
  • Submissions 1, 3 can only match reviewer 2 (capacity 2). Submission 2 can match reviewer 4 (capacity 2).
  • Max flow: reviewer 2 gives 2 (to subs 1 and 3), reviewer 4 gives 2 (to sub 2). Total = 4 < 6.

  • Infeasible. The subset {1, 3} is critical.

    K = 2, remove reviewers {1, 2}:**

  • Remaining reviewers: 3, 4 (both topic 2). No reviewer can cover topic 1 submissions.

  • Max flow for subs 1, 3 = 0. Total = 2 < 6. Infeasible. The subset {1, 2} is critical.

    K = 2, remove reviewers {3, 4}:**

  • Remaining reviewers: 1, 2 (both topic 1). No reviewer can cover topic 2 submission.
  • Max flow for sub 2 = 0. Total = 4 < 6. Infeasible. The subset {3, 4} is critical.

This example shows that with K = 2, many subsets can be critical, and the approach correctly identifies all of them.

General Case: Primary and Secondary Domains

Problem Statement

The general case extends the assignment problem so that:

  • Submissions have both a Primary Submission Domain and a Secondary Submission Domain.
  • Reviewers have both a Primary Domain of Expertise and a Secondary Domain of Expertise.
  • The goal is to find the optimal assignment — not just a feasible one — where "optimal" means maximizing the quality of domain matches (e.g., primary-primary matches are preferred over secondary matches).

This is a more general problem with significantly higher computational complexity when the truly optimal assignment is sought, because it is no longer a pure max-flow problem but a maximum weight bipartite matching** problem.

Proposed Approach

We propose a tiered max-flow approach combined with a min-cost max-flow formulation:

Approach A: Tiered Max-Flow (Greedy Approximation)**

  1. Layer 1 — Primary-Primary matches. Build a flow graph where edges exist only when sub.primary == rev.primary. Run Edmonds-Karp to saturate as many primary-primary assignments as possible. Fix these flows.
  2. Layer 2 — Primary-Secondary matches. On the remaining capacity, build a graph for sub.primary == rev.secondary or sub.secondary == rev.primary. Run max-flow again.
  3. Layer 3 — Secondary-Secondary matches. Finally, attempt sub.secondary == rev.secondary on the still-available capacity.

This greedy tiered approach runs Edmonds-Karp up to 3 times, so its complexity is O(3 × V × E²) = O(V × E²). It is fast but does not guarantee global optimality — a primary-primary match in layer 1 might block a better overall assignment.

Approach B: Min-Cost Max-Flow (Exact Solution)**

  1. Assign a cost to each submission→reviewer edge based on match quality:
    • Primary-Primary: cost = 0 (best)
    • Primary-Secondary: cost = 1
    • Secondary-Secondary: cost = 2 (worst)
  2. Add source→submission edges (capacity = MinReviewsPerSubmission, cost = 0) and reviewer→sink edges (capacity = MaxReviewsPerReviewer, cost = 0).
  3. Run Successive Shortest Path (SSP) min-cost max-flow:
    • While there exists a max-flow augmenting path, choose the one with minimum total cost.
    • Uses Bellman-Ford or Dijkstra (with potentials) for shortest path computation.

The complexity of SSP with Dijkstra and potentials is O(F × (V + E) × log V), where F is the total flow (P × MinReviewsPerSubmission). This guarantees the optimal assignment.

Comparison of Approaches

Approach Complexity Optimality
Tiered Max-Flow (A) O(V × E²) Approximate (greedy)
Min-Cost Max-Flow (B) O(F × (V + E) × log V) Exact (optimal)
Brute-force enumeration O((P×R)!) Exact (impractical)

For typical conference sizes (P < 200, R < 200), Approach B with Dijkstra is practical and produces the optimal assignment. Approach A is a reasonable fallback when speed is critical and near-optimal results suffice.

Input Example

#Submissions
#Id, Title, Authors, E-mail, Primary, Secondary
1, "ML Systems", Alice, alice@x, 1, 2
2, "Security Audit", Bob, bob@x, 3, 1
#Reviewers
#Id, Name, E-mail, Primary, Secondary
1, Rev1, r1@x, 1, 3
2, Rev2, r2@x, 2, 3
3, Rev3, r3@x, 3, 1
#Parameters
MinReviewsPerSubmission, 1
MaxReviewsPerReviewer, 2

Tiered approach (A) result:**

  • Layer 1: Sub 1 (pri=1) matches Rev 1 (pri=1). Sub 2 (pri=3) matches Rev 3 (pri=3). Flow = 2.

  • All assignments are primary-primary. Optimal in this case.

    Min-cost max-flow (B) result:**

  • Same result: Sub 1 ↔ Rev 1 (cost 0), Sub 2 ↔ Rev 3 (cost 0). Total cost = 0.

  • Both approaches agree here because the primary-primary matches are sufficient.

    A case where they diverge:** If Rev 1 could also serve Sub 2 via secondary match (sub.sec=1 == rev.pri=1), the greedy approach assigns Rev 1 to Sub 1 first. But if Sub 2 has no other match and Sub 1 has an alternative, the min-cost approach would globally optimize to ensure both submissions are covered.

Why Two Integration Tests "Differ"

ext_test_diff_overview

Integration tests dataset10 and dataset11 report output differences when compared against the reference solutions. However, our outputs are equally correct — they represent a different but valid decomposition of the same maximum flow.

Input Configuration

Both datasets share the same submissions and reviewers. They differ only in whether risk analysis is enabled (dataset10: off, dataset11: on).

# 3 submissions, all with secondary topic = 2
Submissions: 1 (pri=1, sec=2), 2 (pri=3, sec=2), 3 (pri=4, sec=2)
# 6 reviewers, all with primary topic = 2
Reviewers: 1..6 (all pri=2)
Mode: 2 (secondary submission domain enabled)
MinReviewsPerSubmission = 2, MaxReviewsPerReviewer = 2

In mode 2, a submission matches a reviewer when the reviewer's primary topic equals either the submission's primary or secondary topic. Since all reviewers have primary topic 2, all three submissions match via their secondary topic (2). Every submission can be reviewed by every reviewer.

Total required:** 3 submissions × 2 reviews = 6. Total reviewer capacity:** 6 reviewers × 2 = 12. Plenty of slack.

Our Solution (dataset10)

#SubmissionId,ReviewerId,Match
1, 1, 2 # Sub 1 ← Rev 1 (via topic 2)
1, 2, 2 # Sub 1 ← Rev 2 (via topic 2)
2, 1, 2 # Sub 2 ← Rev 1 (via topic 2)
2, 2, 2 # Sub 2 ← Rev 2 (via topic 2)
3, 3, 2 # Sub 3 ← Rev 3 (via topic 2)
3, 4, 2 # Sub 3 ← Rev 4 (via topic 2)
#Total: 6

Reviewers 1 and 2 each handle 2 reviews (at capacity). Reviewers 3 and 4 each handle 1 review. Reviewers 5 and 6 are unused. Every submission gets exactly 2 reviews.

Reference Solution

#SubmissionId,ReviewerId,Match
1, 1, 2 # Sub 1 ← Rev 1 (via topic 2)
1, 3, 2 # Sub 1 ← Rev 3 (via topic 2)
2, 2, 2 # Sub 2 ← Rev 2 (via topic 2)
2, 3, 2 # Sub 2 ← Rev 3 (via topic 2)
3, 1, 2 # Sub 3 ← Rev 1 (via topic 2)
3, 2, 2 # Sub 3 ← Rev 2 (via topic 2)
#Total: 6

Reviewers 1, 2, and 3 each handle 2 reviews (at capacity). Reviewers 4, 5, 6 are unused. Every submission gets exactly 2 reviews.

Why Both Are Correct

The max-flow problem admits multiple valid flow decompositions that achieve the same total flow. Both solutions:

  • Total flow = 6 (all required reviews assigned)
  • Each submission receives exactly 2 reviews (satisfying MinReviewsPerSubmission)
  • No reviewer exceeds 2 reviews (satisfying MaxReviewsPerReviewer)
  • All edges are valid (every assignment uses the secondary topic 2 match)

The difference arises from the BFS traversal order in Edmonds-Karp. The algorithm finds the shortest* augmenting path at each step, but when multiple shortest paths of equal length exist, the order in which adjacent edges are explored determines which path is chosen. Our adjacency list ordering produces a different — but equally optimal — flow decomposition.

For dataset11 (with risk analysis), the same assignment is used as the baseline. Since no reviewer is individually critical (removing any single reviewer still leaves enough capacity to cover all 6 required reviews), the risk analysis output is empty in both cases — they agree on the risk analysis result, differing only in the assignment listing.

Note
The integration test uses exact diff comparison against a single reference output. This is a limitation of the test harness: it cannot recognize that two different assignments are equally valid. Both outputs represent a correct maximum flow with identical total flow and no constraint violations.
